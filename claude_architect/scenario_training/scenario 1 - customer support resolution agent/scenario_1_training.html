Scenario: You are building a customer support resolution agent using the Claude Agent SDK. The agent handles high-ambiguity requests: returns, billing disputes, and account issues. Your target is 80%+ first-contact resolution (FCR) while knowing when to escalate.

Four MCP tools: get_customer · lookup_order · process_refund · escalate_to_human

The starting problem: The agent ships at ~55% FCR. It escalates simple damage replacements while attempting to autonomously handle policy exceptions. It occasionally skips get_customer and calls lookup_order directly — causing misidentified refunds.

Workflow Enforcement — Programmatic Gates vs Prompts HIGH PRIORITY

The Core Problem

Production data shows the agent skips get_customer in 12% of cases, calling lookup_order directly with only the customer's stated name. This results in misidentified accounts and incorrect refunds. You need this to stop — permanently.

Why Prompts Are Insufficient for This

An instruction like "Always call get_customer before any other tool" works most of the time. But "most of the time" produces real financial errors at scale. Prompt compliance is probabilistic — even perfect prompt engineering leaves a non-zero failure rate.

Enforcement MethodCompliance RateWhen to Use
Prompt instruction~95–99%Style preferences, optional guidance
Few-shot examples~97–99%Consistency for judgment calls
Programmatic gate100%Business-critical sequences with financial/legal consequences
Gotcha — "But We Added Examples"

Adding 5–8 few-shot examples showing get_customer called first raises compliance to ~99%. At 10,000 daily interactions, that's still 100 misidentified refunds per day. The exam knows this. Always choose programmatic enforcement for financial operations.

The Prerequisite Gate Pattern

The gate lives in your tool executor — the code layer that dispatches tool calls from the agent loop. It reads from a session state object, not from conversation history.

# Session state — NOT conversation history
@dataclass
class SessionState:
    verified_customer_id: Optional[str] = None
    total_refunded: float = 0.0
    escalated: bool = False

# Gate runs BEFORE every tool call
def prerequisite_gate(tool_name: str, state: SessionState) -> Optional[dict]:
    if tool_name in ("lookup_order", "process_refund", "escalate_to_human"):
        if state.verified_customer_id is None:
            return {
                "isError": True,
                "errorCategory": "prerequisite_not_met",
                "isRetryable": False,
                "error": f"Cannot call {tool_name}: customer not verified.",
                "required_action": "Call get_customer first."
            }
    return None  # Proceed
Why Not Just Make get_customer Required Input?

You could require customer_id as an input to lookup_order's schema. But the agent could still provide a plausible-looking customer_id it invented. The gate checks that get_customer actually ran and succeeded by reading verified state — not trusting agent-supplied input.

What the Gate Returns When Blocked

The error response becomes a tool result in the conversation. Claude will read it and understand what it needs to do. This is intentional — you're giving the agent a clear, actionable recovery path.

# Agent receives this as the tool result:
{
  "isError": True,
  "errorCategory": "prerequisite_not_met",
  "isRetryable": False,
  "error": "Cannot call lookup_order: customer identity not verified.",
  "required_action": "Call get_customer first to verify customer identity."
}
# Claude reads this, calls get_customer, then retries lookup_order.

Advanced Gotchas Common Mistakes

Gotcha 1 — Multiple Customer Matches

get_customer may return an error if multiple accounts share the same name. Your gate should NOT set verified_customer_id until a unique customer is confirmed. If get_customer errors, state remains None. The gate continues to block. The agent will ask for more info.

Gotcha 2 — State Lives in the Session, Not the Model

If your state is reconstructed from conversation history (e.g., scanning messages for "customer verified"), the gate can be fooled by a message that contains that text for other reasons. State must be an independent Python object populated only by actual tool success.

Gotcha 3 — Session Restart Mid-Workflow

If the application restarts and loads a prior conversation to resume, you must also restore SessionState from a persistent store — not just replay conversation history. If state is lost, all gates reset, forcing re-verification. This is the CORRECT behavior — it's safer to re-verify than to assume prior verification is still valid.

Gotcha 4 — Gate vs. Hook Distinction

The prerequisite gate enforces ordering (A must happen before B). PreToolUse hooks enforce policy (block calls that violate business rules regardless of ordering). Both are deterministic. You need both for a production system.

Full Execution Order in the Tool Executor

def execute_tool(tool_name: str, tool_input: dict, state: SessionState) -> dict:
    # 1. Ordering gate — is it safe to call this tool yet?
    gate_error = prerequisite_gate(tool_name, state)
    if gate_error:
        return gate_error

    # 2. Policy hook — does this call violate a business rule?
    hook_error = pre_tool_use_hook(tool_name, tool_input, state)
    if hook_error:
        return hook_error

    # 3. Execute the actual tool
    result = dispatch[tool_name](**tool_input)

    # 4. PostToolUse — update state, normalize data
    result = post_tool_use_hook(tool_name, result, state)

    return result

Tool Design — Descriptions, Boundaries & Error Responses HIGH PRIORITY

Why Tool Descriptions Matter Most

Tool descriptions are the primary mechanism the model uses for tool selection. Minimal descriptions cause systematic misrouting. This is not a context window issue — it's an information gap issue.

Description QualityEffect
Minimal: "Retrieves customer information"Model confuses with lookup_order; selects wrong tool for order queries
Rich: includes input formats, examples, negative boundariesModel reliably selects correct tool; description IS the routing logic

The Description Blueprint

Every production tool description should answer five questions:

  • What does it do? One sentence, action-oriented.
  • When to use it? Trigger conditions, example queries.
  • What does it accept? Input formats, which fields matter.
  • What does it return? Key output fields.
  • When NOT to use it? Explicit negative boundary — the most important and most skipped.
# BAD — causes get_customer / lookup_order confusion
{
    "name": "get_customer",
    "description": "Retrieves customer information."
}
{
    "name": "lookup_order",
    "description": "Retrieves order details."
}

# GOOD — explicit boundaries eliminate ambiguity
{
    "name": "get_customer",
    "description": (
        "Look up and VERIFY customer identity. REQUIRED first call in every interaction. "
        "Use when: starting any interaction. Input: name + email preferred. "
        "Returns: customer_id, account standing, verification status. "
        "Do NOT use to retrieve order details — use lookup_order for that."
    )
}
{
    "name": "lookup_order",
    "description": (
        "Retrieve details for a specific ORDER. Use when customer references an order number, "
        "shipment, delivery status, or purchase. Input: order_id (format ORD-XXXX). "
        "Returns: item, amount, status, return window eligibility. "
        "REQUIRES get_customer first. Do NOT use for account/identity info."
    )
}

Structured MCP Error Responses Exam Pattern

Generic errors like "Operation failed" prevent intelligent recovery. Structured errors enable the agent to make appropriate decisions about retry, escalation, or clarification.

Error CategoryisRetryableAgent Behavior
transient (timeout, service down)trueRetry with backoff
validation (bad input, not found)falseAsk customer for correct info
business_rule (threshold, policy)falseEscalate to human
permission (access denied)falseInform agent of scope limits
prerequisite_not_metfalseCall required preceding tool
# Every MCP tool should return this shape on error:
{
    "isError": True,
    "errorCategory": "transient",    # transient | validation | business_rule | permission
    "isRetryable": True,
    "error": "Order service timed out after 5 seconds.",
    "action_required": "Retry in a few moments."
}
Gotcha — Empty Result vs. Error

A successful search that returns zero results is NOT an error. Return {"isError": false, "results": [], "count": 0}. If you return an error for "no results found," the agent may retry when there's nothing to retry — or escalate unnecessarily. The coordinator needs to know the difference between "the search failed" and "the search succeeded but found nothing."

Gotcha — System Prompt Keyword Contamination

If your system prompt says "always look up the customer first," the word "look up" may create an association with lookup_order instead of get_customer. Tool descriptions and system prompts interact. Review both together for keyword conflicts.

Escalation Calibration — From 55% to 80%+ FCR HIGH PRIORITY

The Starting Problem

FCR is at 55%. Logs show: escalating simple damage replacements (should resolve) while attempting to handle policy exceptions autonomously (should escalate). The calibration is inverted.

Root Cause: Unclear Decision Boundaries

The agent doesn't have explicit criteria for "when is a case simple enough to resolve?" and "when does a case require human judgment?" It defaults to over-escalation on ambiguity — which sounds safe but destroys FCR and wastes human agent time.

The Fix: Explicit Criteria + Few-Shot Examples

General instructions like "only escalate complex cases" do nothing — they reproduce the ambiguity. You need specific case categories and examples showing the decision in context.

Case TypeActionWhy
Damage/defect + evidenceRESOLVE → process_refundClear policy coverage, no judgment needed
Non-delivery confirmed in systemRESOLVE → process_refundObjective fact, policy covers it
Duplicate charge confirmedRESOLVE → process_refundBilling error, clear entitlement
Customer explicitly requests humanESCALATE IMMEDIATELYCustomer autonomy — do not attempt first
Return window exceededESCALATEPolicy exception = human decision
Refund > $500ESCALATE (blocked by hook)Threshold policy
Policy silent on requestESCALATEDo not interpret or assume policy
Account access issuesESCALATEOutside tool scope
Gotcha 1 — Explicit Human Request

"I need to speak to a real person right now." This must trigger escalate_to_human IMMEDIATELY after get_customer. Do NOT investigate the issue first and then offer to resolve it. That violates customer autonomy and is an exam trap. The customer's preference overrides your FCR metric.

Gotcha 2 — Sentiment ≠ Complexity

Using sentiment analysis to route escalation is a wrong answer. An angry customer with a simple damage replacement should still be resolved autonomously (after acknowledging frustration). A calm customer asking for a policy exception should be escalated. Sentiment and complexity are orthogonal. The exam will offer sentiment-based escalation as a distractor.

Gotcha 3 — Self-Reported Confidence Is Unreliable

Having the agent self-report a confidence score (1–10) and escalate below a threshold sounds reasonable but fails for a specific reason: LLMs are overconfident on hard cases. The agent that incorrectly handles a policy exception will report confidence 8/10 — because it doesn't know what it doesn't know. Confidence scoring is a wrong answer for escalation calibration.

Gotcha 4 — Policy Gap = Escalate, Not Interpret

Customer asks: "Can you match the competitor's price?" Your policy document covers own-site price adjustments but is silent on competitor matching. The WRONG move: interpret policy liberally and process a discount. The CORRECT move: escalate — policy is ambiguous on this specific request. When policy is silent, a human must decide.

The Few-Shot Escalation Examples in System Prompt

ESCALATION FEW-SHOT EXAMPLES:

Example 1 — Standard damage, RESOLVE:
Customer: "My order ORD-5521 arrived with a cracked screen."
Correct: get_customer → lookup_order → process_refund
Reason: Clear damage, within return window, standard policy covers it.

Example 2 — Explicit human request, ESCALATE IMMEDIATELY:
Customer: "I want to speak to a manager right now."
Correct: get_customer → escalate_to_human
Reason: Explicit human request. Do NOT investigate first.

Example 3 — Policy exception, ESCALATE:
Customer: "I want to return this, I bought it 90 days ago."
Correct: get_customer → lookup_order → escalate_to_human
Reason: 90 days exceeds return window. Policy exception = human decision.

Example 4 — Policy gap, ESCALATE:
Customer: "CompetitorX has this for $20 less, can you match it?"
Correct: get_customer → escalate_to_human
Reason: Policy is silent on competitor matching. Do not assume authorization.

Context Management — Preserving Critical Facts Common Exam Area

The Problem: Summarization Loses Numbers

In long customer support sessions, conversation history gets summarized. Summarization compresses narrative well but loses precise values: order numbers, refund amounts, dates, and account statuses get vague or disappear entirely.

Example of Dangerous Summarization

Original: "Customer requested refund of $247.50 for order ORD-4410, confirmed returned on 2024-11-03."

Summarized: "Customer had a return issue that was addressed." — The refund amount, order ID, and return date are gone. A follow-up action in the same session will have no facts to work with.

The Case Facts Block Pattern

Extract critical transactional facts into a persistent structured block that lives OUTSIDE the summarized history. Include it in every subsequent prompt.

# Inject at the TOP of every prompt, before history:
CASE FACTS (authoritative — do not override with summary):
  customer_id: CUST-1007
  customer_name: Tom Rivera
  verified: true
  orders_referenced: [ORD-4410]
  refund_amounts_discussed: [$199.00]
  return_confirmed_date: 2024-11-03
  resolution_status: refund_pending
  escalated: false

Trimming Verbose Tool Output

A real order lookup might return 40+ fields. Only 5 are relevant to refund resolution. Verbose tool output accumulates in context and pushes important information toward the middle of the window — where attention is lowest.

# PostToolUse hook — trim before model sees it
def post_tool_use_hook(tool_name: str, result: dict, state: SessionState) -> dict:
    if tool_name == "lookup_order":
        # Keep ONLY what matters for support resolution
        relevant_fields = ["order_id", "item", "amount", "status",
                           "days_since_order", "within_return_window", "isError"]
        return {k: result[k] for k in relevant_fields if k in result}
    return result

Lost-in-the-Middle Effect

Models reliably process information at the beginning and end of long inputs, but miss content in the middle. For multi-issue sessions with multiple tool results:

  • Put key facts summary at the beginning
  • Put current task context at the end
  • Use section headers to help the model navigate middle content

Multi-Issue Decomposition Exam Pattern

Customer: "My order ORD-9901 arrived damaged AND I was billed the wrong amount last month."

The agent should decompose this into two parallel investigations, then synthesize a single response — NOT handle them sequentially in two separate exchanges.

# Agent decomposes and addresses both in one pass:
Issue 1: Damaged order ORD-9901
  → lookup_order(ORD-9901) → process_refund($85.00, "damaged on arrival")

Issue 2: Billing discrepancy
  → lookup_order(billing lookup) → process_refund($44.00, "billing error")

# Then synthesizes ONE response covering both resolutions

Hooks — PreToolUse & PostToolUse Exam Pattern

HookFiresReturns blocked error?Use Case
PreToolUseBefore tool executesYes — intercepts callPolicy enforcement, input validation, compliance blocking
PostToolUseAfter tool returnsNo — transforms resultData normalization, state updates, field trimming, enrichment

PreToolUse: The $500 Threshold

def pre_tool_use_hook(tool_name: str, tool_input: dict, state: SessionState) -> Optional[dict]:
    if tool_name == "process_refund":
        amount = tool_input.get("amount", 0)
        if amount > 500:
            return {
                "isError": True,
                "errorCategory": "business_rule",
                "isRetryable": False,
                "error": f"Refund of ${amount:.2f} exceeds the $500 automated threshold.",
                "action_required": "Use escalate_to_human for manager approval."
            }
    return None  # Proceed normally

PostToolUse: Format Normalization

Different backend services return dates in different formats (Unix timestamps, ISO 8601, human-readable strings). A PostToolUse hook normalizes all of them before the model processes the result, preventing format confusion in reasoning.

def normalize_dates(result: dict) -> dict:
    # Convert any Unix timestamp to ISO 8601
    for field in ("created_at", "shipped_at", "delivered_at"):
        if field in result and isinstance(result[field], (int, float)):
            result[field] = datetime.utcfromtimestamp(result[field]).isoformat()
    return result
Hook vs. Prerequisite Gate vs. Prompt — Decision Rule

Hooks = intercept specific tool calls for policy/normalization. Prerequisite gates = enforce ordering of tool calls. Prompts = guidance for judgment calls, style, escalation criteria. When you need 100% compliance on a business rule, use hooks or gates. When you need good judgment, use prompts + few-shot.

Labs — Hands-On Exercises

Lab 1 — Verify the Prerequisite Gate

Observe the gate blocking and self-correcting agent behavior.

  1. Run python test_runner.py --dry-run and read all 12 cases.
  2. In agent.py, temporarily comment out the gate in execute_tool() — the line that calls prerequisite_gate().
  3. Run python test_runner.py --case C001. Does the agent still call get_customer first?
  4. Modify the system prompt to remove all references to get_customer. Run again. Note the change in behavior without the gate.
  5. Restore the gate. Confirm the agent always calls get_customer first regardless of system prompt wording.
  6. Observation to note: With the gate, even a degraded system prompt results in correct ordering.

Lab 2 — Test the $500 PreToolUse Hook

Observe policy enforcement at the hook layer.

  1. Run python test_runner.py --case C011 (the $750 refund case).
  2. Observe the hook blocking the process_refund call and the agent redirecting to escalate_to_human.
  3. Temporarily change the threshold from 500 to 1000 in pre_tool_use_hook(). Re-run C011. What happens?
  4. Restore the threshold. Now remove the hook entirely and re-run C011. Does the system prompt alone prevent the $750 refund?
  5. Key insight: How often does the model attempt the refund when the hook is absent?

Lab 3 — Escalation Calibration

Tune the escalation criteria to hit 80%+ FCR.

  1. Remove the few-shot escalation examples from SYSTEM_PROMPT in agent.py, keeping only the criteria table.
  2. Run python test_runner.py. Record the FCR rate.
  3. Add back the few-shot examples. Run again. Compare FCR.
  4. Run python test_runner.py --case C004 (explicit human request). Observe whether the agent investigates first or escalates immediately.
  5. Run python test_runner.py --case C006 (competitor price match). Confirm the agent escalates due to policy gap — not because it's complex.

Lab 4 — Multi-Match Clarification (C009)

Verify the agent asks for more info instead of guessing.

  1. Run python test_runner.py --case C009.
  2. Verify the agent asks for email or order number — it should NOT pick one of the matching customers heuristically.
  3. In mcp_get_customer(), change the multi-match response to return the first match silently. Re-run C009. This simulates the wrong behavior — note how it could process a refund for the wrong customer.
  4. Restore correct behavior.
