Agentic Architecture & Orchestration

Exam Weight: 27% — Highest Priority

An agent is not just a model that answers questions — it is a model that acts: calling tools, observing results, reasoning about what to do next, and continuing until the task is complete. This domain tests whether you can diagnose and fix broken agentic systems.

The exam will present broken or poorly designed agentic systems and ask you to diagnose and fix them. Surface knowledge will not pass this exam.

The Agentic Loop

What It Is

The agentic loop is the heartbeat of every agent — the cycle of send → inspect stop_reason → execute tool → append result → repeat.

stop_reasonMeaningYour Action
"tool_use"Claude wants to call a toolExecute, append result, loop again
"end_turn"Claude is finishedReturn the response to the user

The Correct Loop in Python

def run_agent(user_message: str, tools: list) -> str:
    messages = [{"role": "user", "content": user_message}]

    while True:
        response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=4096,
            tools=tools,
            messages=messages
        )

        if response.stop_reason == "end_turn":
            for block in response.content:
                if hasattr(block, "text"):
                    return block.text

        elif response.stop_reason == "tool_use":
            messages.append({"role": "assistant", "content": response.content})

            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    result = execute_tool(block.name, block.input)
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result
                    })

            messages.append({"role": "user", "content": tool_results})
            # Loop continues

Anti-Patterns (Will Be on the Exam)

Anti-Pattern 1 — Natural Language Termination

if "I have completed" in response.text: break — Claude might say "I have completed the first step" mid-task. Use stop_reason.

Anti-Pattern 2 — Iteration Cap as Primary Stop

for i in range(10): response = call_claude() — Complex tasks may need more iterations. A safety cap alongside stop_reason is OK; as primary mechanism it is not.

Anti-Pattern 3 — Content Type as Completion

if response.content[0].type == "text": return — Claude can return mixed text + tool_use in one response. Always check stop_reason.

Anti-Pattern 4 — Not Appending Tool Results

Executing a tool but not adding its result to messages. Claude has no memory between API calls — it will repeat the call or hallucinate.

                │   COORDINATOR AGENT │
                │  - Decomposes task  │

                           │
          ┌────────────────┼────────────────┐
          │                │                │
┌─────────▼──────┐ ┌───────▼──────┐ ┌──────▼───────┐
│  WEB SEARCH    │ │   DOCUMENT   │ │  SYNTHESIS   │
│    AGENT       │ │   ANALYSIS   │ │    AGENT     │
└────────────────┘ └──────────────┘ └──────────────┘

Critical Rules

  • Subagents have isolated context — they do NOT inherit the coordinator's history. Pass everything explicitly.
  • All communication routes through the coordinator — subagents never talk to each other directly.
  • The coordinator owns task decomposition — narrow decomposition produces correct but incomplete output.
  • The coordinator owns error handling — retry, use partial results, skip and annotate, or escalate.

Iterative Refinement

         ↓

         ↓

1.3

Subagent Invocation, Context Passing & Spawning

Context Injection — Most Common Exam Topic

Wrong — Implicit Context

invoke_subagent("synthesis_agent", {"instruction": "Synthesize the research findings"}) — What findings? The subagent has no idea.

Correct — Explicit Context

Pass web_search_results, document_analysis_results, source_metadata, research_query, and quality_criteria all explicitly in the subagent prompt.

Parallel Subagent Spawning

Emit multiple Task tool calls in a single coordinator response — not across separate turns. Sequential spawning wastes time for independent tasks.

Coordinator Prompt Design

TypeExample
Weak (procedural)"Step 1: Call web_search with query X. Step 2: Call doc_analysis..."
Strong (goal-oriented)"Research comprehensively. Ensure coverage spans all subtopics. Every claim must trace to a source."

def execute_tool(tool_name: str, tool_input: dict) -> dict:
    if tool_name == "lookup_order":
        if state.verified_customer_id is None:

1.5

Agent SDK Hooks

HookFires WhenUse Case
PreToolUseBefore a tool call executesBlock policy violations, validate inputs
PostToolUseAfter a tool call returnsNormalize data formats, enrich results
Hooks vs. Prompts — Exam Rule

Use hooks for: financial thresholds, identity verification before sensitive ops, compliance requirements, any rule needing 100% enforcement. Use prompts for: style preferences, guidelines, rules where occasional non-compliance is acceptable.

Task Decomposition Strategies

PatternUse WhenBest For
Prompt ChainingPredictable structure, each step output → next step input, sequence never changesCode review pipelines, document processing
Dynamic AdaptiveOpen-ended, intermediate findings determine next steps, full plan unknown upfrontCodebase investigation, debugging, open research

The Code Review Split Pattern

Step 1: Per-file local analysis (N parallel calls, one per file)
         → Each call sees only one file → focused, deep analysis

Step 2: Cross-file integration pass (one call with all per-file summaries)
         → Data flow bugs, interface mismatches, architectural violations

When Resuming After Code Changes

Tell the agent explicitly what changed. Don't just resume and ask the same question — Claude will reason from stale tool results.
