Lab D2-01

Tool Design & MCP Configuration

Domain 2 — Tool Use & Integration Task Statements 2.1, 2.2, 2.3 ⏱ 45 minutes

Objective: Design Claude-facing tool descriptions that eliminate ambiguity and prevent misuse. You'll write poor descriptions, observe how they cause selection errors, then apply the 5-element framework to fix them. You'll also wire up a complete tool call and validate schema design against the CCAF exam standard.

Key Insight: Tool descriptions are the interface between Claude's reasoning and your infrastructure. Vague descriptions cause wrong tool calls. Missing negative boundaries cause dangerous calls on edge cases. No exam tests if you know tool names — they test if you can design them correctly.

Part 1

Diagnose Bad Tool Descriptions (15 min)

These tool descriptions are broken. Before looking at the fixes, identify which element is missing: what, when, inputs, outputs, negative boundary.

Tool A — Missing Negative Boundary

❌ Broken
search_knowledge_base:
  "Searches the internal
   knowledge base for
   relevant articles."
✅ Fixed
search_knowledge_base:
  "Searches internal KB for
   policy/procedure articles.
   Use for: policy questions,
   procedure lookups. Do NOT
   use for: customer account
   data, order history, or
   real-time inventory."

Without the negative boundary, Claude may call search_knowledge_base when asked about a customer's order history — wasting a call and returning confusing empty results.

Tool B — Missing 'When' Condition

❌ Broken
escalate_to_human:
  "Escalates the case to a
   human support agent."
✅ Fixed
escalate_to_human:
  "Transfers conversation to
   a human agent. Use ONLY
   when: (1) customer
   explicitly requests human,
   (2) refund > $500 requires
   manager approval, (3) 3+
   failed resolution attempts.
   Do NOT use preemptively."

Part 2

Schema Design — Input Validation Shapes (15 min)

Claude's tool call inputs are validated against the schema you provide. Poorly designed schemas cause: (a) Claude sending wrong types, (b) Claude omitting required fields, (c) ambiguous optional fields with no defaults.

Reference Schema — process_refund

tools = [
  {
    "name": "process_refund",
    "description": "Initiates a refund for a delivered order. Use when a customer requests a refund for a verified order. Requires prior successful lookup_order call. Do NOT use for exchanges or replacements — use create_exchange_request for those.",
    "input_schema": {
      "type": "object",
      "properties": {
        "order_id": {
          "type": "string",
          "description": "Order ID from the verified lookup_order result. Format: ORD-XXXX."
        },
        "amount": {
          "type": "number",
          "description": "Refund amount in USD. Must be <= order amount. Must be <= 500 for automated processing.",
          "minimum": 0.01,
          "maximum": 500
        },
        "reason": {
          "type": "string",
          "description": "Customer-stated reason for refund.",
          "enum": ["damaged", "wrong_item", "not_received", "other"]
        }
      },
      "required": ["order_id", "amount", "reason"]
    }
  }
]

Identify the Anti-Patterns

Anti-PatternWhat Goes WrongFix
No minimum/maximumClaude may send amount: -50 or amount: 9999Add numeric bounds
No enum for reasonClaude invents freeform values like "product was bad"Enumerate valid reasons
Required without descriptionClaude infers the field's meaning from the name aloneAlways add per-field description
Optional field with no defaultClaude may omit it, causing API 400 errorsMake it required OR set a default
Exam Insight

The exam tests whether you know: maximum: 500 in the schema does not replace the PreToolUse hook. Schema validation catches malformed inputs. The hook enforces business rules. Both layers are needed — they do different jobs.

Part 3

Tool Selection Simulation (15 min)

# Run this to see how description quality affects tool selection
import anthropic

client = anthropic.Anthropic()

VAGUE_TOOLS = [
    {"name": "search", "description": "Searches for information.",
     "input_schema": {"type": "object", "properties": {"query": {"type": "string"}}, "required": ["query"]}},
    {"name": "lookup", "description": "Looks up records.",
     "input_schema": {"type": "object", "properties": {"id": {"type": "string"}}, "required": ["id"]}},
]

CLEAR_TOOLS = [
    {"name": "search_knowledge_base",
     "description": "Searches internal policy and procedure articles. Use for policy questions. Do NOT use for customer account data or order history.",
     "input_schema": {"type": "object", "properties": {"query": {"type": "string", "description": "Policy search terms"}}, "required": ["query"]}},
    {"name": "lookup_order",
     "description": "Retrieves order details for a verified customer. Use for order-specific questions. Requires prior get_customer call. Do NOT use for policy lookups.",
     "input_schema": {"type": "object", "properties": {"order_id": {"type": "string", "description": "Order ID in format ORD-XXXX"}}, "required": ["order_id"]}},
]

def test_selection(tools, label, prompt):
    response = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=512,
        tools=tools,
        messages=[{"role": "user", "content": prompt}]
    )
    for block in response.content:
        if block.type == "tool_use":
            print(f"[{label}] Selected: {block.name} | Input: {block.input}")

test_selection(VAGUE_TOOLS, "VAGUE", "What is the return policy for damaged items?")
test_selection(CLEAR_TOOLS, "CLEAR", "What is the return policy for damaged items?")
test_selection(VAGUE_TOOLS, "VAGUE", "Get details for order ORD-5001")
test_selection(CLEAR_TOOLS, "CLEAR", "Get details for order ORD-5001")
Q

Reflection Questions

1. In the vague-tools test, did Claude select the wrong tool for "return policy" or "order ORD-5001"? Why was it ambiguous?
Both search and lookup are plausible with vague descriptions. Specificity eliminates guesswork.
2. The schema includes maximum: 500 for the refund amount. Does this replace the PreToolUse hook? Why or why not?
Schema validation and business-rule enforcement serve different purposes.
3. What does a negative boundary ("Do NOT use for X") communicate to Claude that a positive description alone cannot?
Think about edge cases where Claude's reasoning is ambiguous between two tools.
4. You have a get_customer tool and a search_customer tool. How do you distinguish when Claude should call each one?
One looks up by ID (known identity). The other searches by name (unknown or partial identity).
5. If a tool returns empty results (not an error), what should the structured response look like? How is it different from an error?
isError: false, results: [] is not an error. The same generic error for different failure types prevents recovery.
