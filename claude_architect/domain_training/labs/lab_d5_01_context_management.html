Lab D5-01

Context Management

Domain 5 — Context Management Task Statements 5.1, 5.2, 5.3 ⏱ 40 minutes

Objective: Implement and compare three context management strategies: rolling window, tool result trimming, and structured summarization. Measure the token budget consumed by each approach and observe how they affect reasoning quality on long customer support conversations. Domain 5 is 15% of the exam.

Key Insight: Every API call sends the entire conversation history. Context grows unbounded unless you actively manage it. Poor context management causes: (a) token limit errors, (b) precision value loss from summarization, (c) degraded reasoning on long conversations. The solution is never to just truncate — it's to preserve what matters.

Part 1

Token Budget Measurement (10 min)

Before managing context, measure it. Count tokens to understand your budget and where you're spending it.

import anthropic, json
from typing import List

client = anthropic.Anthropic()

# Claude claude-opus-4-5 context window: 200K tokens
# Practical limit: leave ~20K for output = 180K for input
MAX_CONTEXT_TOKENS = 180_000

def count_message_tokens(messages: List[dict], system: str = "") -> int:
    # Estimate: 1 token ≈ 4 chars (rough approximation)
    total_chars = len(system)
    for msg in messages:
        content = msg.get("content", "")
        if isinstance(content, list):
            for block in content:
                if isinstance(block, dict):
                    total_chars += len(json.dumps(block))
        else:
            total_chars += len(str(content))
    return total_chars // 4  # approximate

# Simulate a long 20-turn conversation
def build_long_conversation(turns: int = 20) -> List[dict]:
    messages = []
    for i in range(turns):
        messages.append({"role": "user", "content": f"Turn {i+1}: Customer message with some details about their order ORD-500{i} and asking about refund status."})
        messages.append({"role": "assistant", "content": f"Turn {i+1}: Agent response explaining the status of order ORD-500{i} and next steps for the refund process."})
    return messages

messages = build_long_conversation(20)
tokens = count_message_tokens(messages)
print(f"20-turn conversation: ~{tokens:,} tokens ({tokens/MAX_CONTEXT_TOKENS:.1%} of budget)")
Part 2

Three Strategies — Implement & Compare (20 min)

Strategy 1: Rolling Window (Keep Last N Turns)

def rolling_window(messages: List[dict], max_turns: int = 10) -> List[dict]:
    # Keep the first message (may contain original request) + last N turns
    if len(messages) <= max_turns * 2:
        return messages
    # Always keep first user message (original context)
    first = messages[:1]
    recent = messages[-(max_turns * 2):]
    return first + recent

# Tradeoff: Simple and cheap, but drops middle conversation turns
# Risk: If critical info was stated in dropped turns, it's gone

Strategy 2: Tool Result Trimming

def trim_tool_results(messages: List[dict], max_chars: int = 500) -> List[dict]:
    trimmed = []
    for msg in messages:
        if isinstance(msg.get("content"), list):
            new_content = []
            for block in msg["content"]:
                if isinstance(block, dict) and block.get("type") == "tool_result":
                    # Trim verbose tool output — keep key fields only
                    content_str = str(block.get("content", ""))
                    if len(content_str) > max_chars:
                        block = block.copy()
                        block["content"] = content_str[:max_chars] + "... [trimmed]"
                new_content.append(block)
            trimmed.append({**msg, "content": new_content})
        else:
            trimmed.append(msg)
    return trimmed

# Tradeoff: Keeps all turns, reduces verbose API responses
# Risk: Trimmed portions may contain error details needed for diagnosis

Strategy 3: Structured Summarization (Preserve Precision)

def summarize_with_pinned_facts(messages: List[dict], facts: dict, keep_last_n: int = 4) -> List[dict]:
    """
    Compress old turns into a structured summary.
    CRITICAL: Precision values (order IDs, amounts) go in facts block, NOT the summary.
    The summary captures intent. The facts block captures exact values.
    """
    if len(messages) <= keep_last_n * 2:
        return messages

    old_turns = messages[:-(keep_last_n * 2)]
    recent_turns = messages[-(keep_last_n * 2):]

    # Summarize old turns (intent only, no precision values)
    summary_content = "[CONVERSATION SUMMARY]\n"
    summary_content += f"Covered {len(old_turns)//2} earlier turns. Customer contacted support about a damaged order.\n"
    summary_content += "Agent verified customer identity and looked up order details.\n"
    summary_content += "[Precision values pinned in system prompt case_facts block - do not re-extract from this summary]\n"

    compressed = [
        {"role": "user", "content": summary_content},
        {"role": "assistant", "content": "Understood. I have the case history and will continue from here."}
    ]

    return compressed + recent_turns

# Key insight: Summary has INTENTIONALLY vague amounts/IDs
# because exact values live in the case_facts block in the system prompt

Part 3

Multi-Agent Context Isolation (10 min)

This is the most-tested context management concept in the exam: subagents have zero inherited context. When an orchestrator spawns a subagent, the subagent starts with an empty conversation history. Nothing from the orchestrator's session is automatically available.

Most Common Multi-Agent Bug

Orchestrator verified the customer and discovered order details. It spawns a refund subagent. The subagent has no knowledge of the verified customer ID or order details — they were never passed. Result: the subagent either errors or re-prompts the customer.

def spawn_refund_subagent_correct(facts: dict, orchestrator_state: dict) -> str:
    # CORRECT: Pass ALL required context explicitly in the subagent's prompt
    subagent_prompt = f"""You are a refund processing specialist.

CASE CONTEXT (verified by orchestrator — do not re-verify):
- Customer ID: {facts['customer_id']} (VERIFIED)
- Order ID: {facts['order_id']} (VERIFIED, belongs to customer)
- Refund Amount: ${facts['refund_amount']:.2f}
- Issue Type: {facts['issue_type']}
- Policy: Refunds under $500 can be processed automatically.

Your task: Process this refund and confirm completion.
Do not ask for customer ID or order ID — they are already verified."""

    response = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=512,
        system=subagent_prompt,  # Context is in system prompt, not inherited
        messages=[{"role": "user", "content": "Process the refund."}]
    )
    return response.content[0].text

def spawn_refund_subagent_broken() -> str:
    # BROKEN: No context passed — subagent is blind
    response = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=512,
        messages=[{"role": "user", "content": "Process the refund."}]
    )
    return response.content[0].text  # Will ask: "Which order? Which customer?"
Q

Reflection Questions

1. The rolling window strategy drops turns 1-10 of a 20-turn conversation. What information is at risk of being lost?
Original customer issue statement, first verified identity, initial order ID — anything mentioned early in the conversation.
2. In structured summarization, why does the summary intentionally NOT include specific amounts and order IDs?
Those values should only be read from the case_facts block. Having them in both places creates a contradiction risk.
3. An orchestrator spawns 3 parallel subagents. Each needs: customer name, verified order list, and current session policies. How do you pass this without the subagents re-fetching everything?
Serialize the verified state object to JSON and embed it in each subagent's system prompt explicitly.
4. A tool result returns 10,000 characters of inventory data. What's the right trimming approach?
Don't trim blindly. Extract only the fields Claude needs from the result and discard the rest before it enters the context.
5. What's the difference between context window exhaustion and context drift? How does each degrade agent performance?
Exhaustion = hard token limit hit → API error. Drift = precision values get paraphrased/lost in long histories → silent accuracy loss.
