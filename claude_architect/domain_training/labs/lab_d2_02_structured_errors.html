Lab D2-02

Structured Error Responses

Domain 2 — Tool Use & Integration Task Statements 2.3, 2.4 ⏱ 40 minutes

Objective: Build the MCP-compatible error response schema and observe how structured errors enable Claude to self-correct, while unstructured errors cause retry loops and dead ends. Every CCAF scenario exercises error handling — this lab locks in the pattern.

Core pattern: Every tool result must include isError (bool), errorCategory (transient/validation/business_rule/permission), and isRetryable (bool). Empty results are not errors.

Part 1

Error Category Reference & isRetryable Logic (15 min)

errorCategoryExampleisRetryableClaude's Next Action
transientPayment gateway timeoutTrueWait and retry automatically
validationOrder ID format invalidFalseAsk customer to re-confirm order number
business_ruleRefund exceeds $500 limitFalseEscalate to human for approval
permissionAgent not authorized for this actionFalseExplain limitation, offer alternatives
not_foundOrder ID doesn't existFalseAsk customer to verify the ID
Critical: Empty Result ≠ Error

{"isError": false, "results": []} means "search succeeded, nothing matched." This is a success. Claude should tell the customer no results were found. If you return isError: true for empty results, Claude will attempt to retry a query that's working correctly.

def make_error(category: str, message: str, retryable: bool, action: str = None) -> dict:
    result = {
        "isError": True,
        "errorCategory": category,
        "isRetryable": retryable,
        "error": message,
    }
    if action:
        result["required_action"] = action
    return result

# Usage examples
timeout_error = make_error(
    "transient",
    "Payment API timeout after 5s.",
    retryable=True
)

threshold_error = make_error(
    "business_rule",
    "Refund $749 exceeds $500 automated limit.",
    retryable=False,
    action="Call escalate_to_human for manager approval."
)

not_found = make_error(
    "not_found",
    "Order ORD-9999 not found in system.",
    retryable=False,
    action="Ask customer to verify order ID."
)

# Empty result — this is NOT an error
empty_search = {"isError": False, "results": [], "message": "No articles matched your search."}
Part 2

Generic vs Specific Error — Self-Correction Test (15 min)

Run two versions of the same error scenario and observe whether Claude self-corrects. Generic errors give Claude no information about what action to take next. Structured errors give Claude the exact next step.

import anthropic, json

client = anthropic.Anthropic()

# Simulate the tool returning two different error formats
GENERIC_ERROR = {"error": "Refund failed."}
STRUCTURED_ERROR = {
    "isError": True,
    "errorCategory": "business_rule",
    "isRetryable": False,
    "error": "Refund $749 exceeds the $500 automated limit.",
    "required_action": "Call escalate_to_human for manager approval."
}

def simulate_error_handling(error_response: dict, label: str):
    # Step 1: initial refund attempt
    first_call = [
        {"role": "user", "content": "Process a $749 refund for order ORD-5002. Customer C001 is verified."},
        {"role": "assistant", "content": [{
            "type": "tool_use", "id": "tool_1",
            "name": "process_refund",
            "input": {"order_id": "ORD-5002", "amount": 749, "reason": "wrong_item"}
        }]},
        {"role": "user", "content": [{
            "type": "tool_result", "tool_use_id": "tool_1",
            "content": json.dumps(error_response),
            "is_error": error_response.get("isError", "error" in error_response)
        }]}
    ]
    response = client.messages.create(
        model="claude-opus-4-5", max_tokens=512,
        tools=[escalate_tool],  # Only escalate_to_human available
        messages=first_call
    )
    print(f"\n=== [{label}] Claude's response after error ===")
    for block in response.content:
        if hasattr(block, "text"): print("Text:", block.text[:200])
        if block.type == "tool_use": print("Tool called:", block.name)

simulate_error_handling(GENERIC_ERROR, "GENERIC")
simulate_error_handling(STRUCTURED_ERROR, "STRUCTURED")
Expected Outcome

With the generic error, Claude may retry the refund or output a vague apology. With the structured error + required_action, Claude should call escalate_to_human. That's self-correction driven by error design.

Part 3

The isRetryable Decision — get_customer Timeout (10 min)

This was one of the deferred knowledge check questions. Settle it with code and reasoning.

Scenario: get_customer times out

The customer database is experiencing high load. The tool call fails with a 5-second timeout. Should isRetryable be True or False?

# Correct: transient timeout IS retryable
timeout_response = {
    "isError": True,
    "errorCategory": "transient",
    "isRetryable": True,
    "error": "Customer database timed out after 5s.",
    "retryAfterMs": 2000  # Optional: hint to wait 2s before retry
}

# Why isRetryable matters:
# If isRetryable=True: orchestrator can auto-retry without bothering the customer
# If isRetryable=False: orchestrator escalates or asks customer to try again
# Using False for a timeout = unnecessary customer friction
# Using True for a business_rule violation = infinite retry loop

# WRONG: validation error marked as retryable
# bad_response = {
#     "isError": True,
#     "errorCategory": "validation",  # <-- permanent failure
#     "isRetryable": True,            # <-- CONTRADICTION — will loop forever
#     "error": "Invalid order ID format: order123"
# }
ScenarioisRetryableReason
Database timeoutTrueInfrastructure issue — will resolve
Invalid order ID formatFalsePermanent — retrying with same input won't help
$750 refund over thresholdFalseBusiness rule violation — amount won't change
Payment API rate limitedTrueTemporary — retry after delay
Insufficient permissionsFalseWon't resolve without authorization change
