Domain 5

Context Management & Reliability

Exam Weight: 15%

Context is Claude's working memory. Everything it knows in a session lives in the context window — and that window is finite. This domain tests whether you can design systems that preserve critical information, escalate correctly, propagate errors intelligently, and build human review workflows that scale.

The exam presents systems that are losing information, escalating incorrectly, or silently failing — and asks you to diagnose and fix them.

Scenario 1: Customer Support Scenario 3: Multi-Agent Research Scenario 6: Structured Data Extraction
5.1

Manage Conversation Context

Three Ways Critical Information Gets Destroyed

1
Progressive Summarization
Original: "Customer paid $1,247.83 on March 15th for order #ORD-98234"

After summarization: "Customer had a payment issue last month"

Amount, date, and order ID are gone — cannot be recovered.
2
Lost in the Middle
Models reliably process information at the beginning and end of long contexts. In a 40-source report, sources 15–25 may be underrepresented in the synthesis.
3
Tool Result Accumulation
Every tool call appends a result to context. An order lookup returning 40 fields (most irrelevant) consumes significant tokens for every subsequent call.

The Case Facts Pattern

Extract transactional facts into a persistent block outside the summarized history, placed at the beginning of every prompt (high-attention position).

=== CASE FACTS (DO NOT LOSE THESE) ===
Customer ID: CUST-00123
Customer: Jane Smith
Orders: ORD-98234, ORD-99011
Amounts: {"ORD-98234": 1247.83, "ORD-99011": 89.00}
Status: refund_in_progress
=====================================

Position-Aware Input Organization

Start (↑ attn)
Key findings summary — sets context correctly, read first
Middle (↓ attn)
Detailed source findings — supporting information, lower attention is fine
End (↑ attn)
Synthesis requirements / instructions — Claude follows these carefully

Trimming Verbose Tool Outputs

Before appending a tool result to the conversation, trim it to relevant fields. A 40-field order record reduced to 8 relevant fields saves significant context for every subsequent API call.

5.2

Escalation and Ambiguity Resolution

When to Escalate — The Correct Triggers

Always Escalate
  • Customer explicitly requests a human agent
  • Policy is silent/ambiguous on this specific request
  • Cannot make meaningful progress after genuine attempts
Consider Escalating
  • High-value transaction with unusual circumstances
  • Customer has made the same complaint 3+ times
Do NOT Escalate For
  • Customer sentiment / frustration level alone
  • Your own confidence score
  • General "complexity" — many complex-seeming requests are routine
Critical Rule

If a customer explicitly asks for a human — escalate immediately, even if you could resolve it. Honor the request without attempting resolution first.

Handling Ambiguous Customer Matches

When get_customer returns multiple matches, never select based on heuristics (most recent, highest-value, etc.). Always ask for additional identifiers: email, order number, or last 4 digits of payment method.

5.3

Error Propagation Across Multi-Agent Systems

The Structured Error Contract

Every error a subagent returns must enable the coordinator to make a decision. {"error": "search failed"} gives the coordinator nothing to work with.

def create_structured_error(error_type, message, attempted,
                               partial_results=None, alternatives=None,
                               is_retryable=True) -> dict:
    return {
        "isError": True,
        "errorType": error_type,       # transient/validation/permission/business
        "message": message,            # Human-readable description
        "attemptedAction": attempted,  # What was tried
        "isRetryable": is_retryable,   # Should coordinator retry?
        "partialResults": partial_results or [],   # Results before failure
        "suggestedAlternatives": alternatives or [],  # Other approaches
        "timestamp": datetime.utcnow().isoformat()
    }

Coverage Annotations in Synthesis

When a subagent fails, the synthesis output must mark which areas lack coverage. Coordinators and end users need to know what's well-supported vs. uncertain.

{
    "synthesis": generate_synthesis(findings),
    "coverage_map": {
        "remote_work_trends": {"status": "well-covered", "confidence": "high"},
        "urban_vacancy_rates": {"status": "incomplete", "confidence": "low",
                                  "reason": "Search API timed out"}
    },
    "note": "Topics in coverage_gaps had source failures. Treat claims with caution."
}
5.4

Context in Large Codebase Exploration

Scratchpad Files

When context fills up during extended exploration, scratchpad files persist findings across context boundaries and survive context window limits.

# Instruct Claude:
"As you explore the codebase, maintain a scratchpad at .claude/scratchpad.md.
After each investigation phase, summarize key findings there. Before starting
each new phase, read the scratchpad to refresh your memory of prior findings."

Crash Recovery with Manifests

For long multi-agent workflows that might fail partway through, each agent exports its state to a known location after completing. On resume, load the manifest and inject prior agent states into prompts — agents that completed don't re-run.

The /compact Command

Use /compact during extended exploration to reduce context usage when verbose discovery output accumulates. Claude summarizes the conversation before continuing, freeing context for more exploration.

When to /compact

Context window is >70% full during multi-phase exploration · You've completed one phase and are starting another · Verbose tool outputs are dominating the context

5.5

Human Review Workflows and Confidence Calibration

The Aggregate Accuracy Trap

Overall 97% accuracy sounds good — until you break it down by segment:

Invoices from Vendor A
99.5%
Invoices from Vendor B
81%
Overall (masks Vendor B)
97%

Always validate accuracy by document type AND field before automating.

Stratified Random Sampling

Don't audit only low-confidence extractions

You'll miss errors in extractions Claude was incorrectly confident about. Always sample high-confidence extractions too — catch miscalibrated confidence.

# All low-confidence → human review
review_queue.extend([e for e in extractions
    if e.get("extraction_confidence") == "low"])

# Random 5% sample of HIGH-confidence → catch miscalibrated confidence
high_confidence = [e for e in extractions
    if e.get("extraction_confidence") == "high"]
review_queue.extend(random.sample(high_confidence, sample_size))

# All extractions from historically high-error vendors → always review
review_queue.extend([e for e in extractions
    if e.get("vendor_name") in high_risk_vendors])

Field-Level Confidence Routing

Schema-level confidence is too coarse. Field-level confidence lets you route review attention precisely:

Low-confidence fieldsRouting
0 fieldsauto_process
1 fieldspot_check — human reviews just that field
2+ fieldsfull_review — human reviews entire extraction
5.6

Information Provenance in Multi-Source Synthesis

Claim-Source Mappings Must Be Preserved

Attribution Lost in Summarization

"Remote work has caused urban office vacancies to rise" — Which source? Publication date? Methodology? All lost. Cannot be recovered.

# RIGHT: Structured claim-source mapping
{
    "claim": "Office vacancy rates in major CBDs hit 22% in Q4 2023",
    "source_url": "https://...",
    "source_title": "Remote Work Exodus: Cities Losing Downtown Tenants",
    "publication_date": "2024-02-15",
    "confidence": "high",
    "excerpt": "office vacancy rates hit 22% in Q4 2023, highest since 2008"
}

Handling Conflicting Statistics

When two credible sources report different numbers, do NOT pick one. Annotate both with attribution and let the coordinator or human decide:

{
    "conflict_detected": True,
    "value_a": {"value": "18%", "source": "...", "date": "2023-03-01"},
    "value_b": {"value": "22%", "source": "...", "date": "2023-12-01"},
    "possible_explanation": "Sources are 9 months apart — values may reflect different time periods",
    "resolution": "human_review_required"
}
Temporal Data — Always Require Dates

Without dates: "18% vacancy" vs "22% vacancy" looks like a contradiction. With dates: Q1 2023 vs Q4 2023 reveals a trend. What looks like a conflict may just be data from different time periods.
