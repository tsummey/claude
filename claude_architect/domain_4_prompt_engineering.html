Prompt Engineering & Structured Output

Exam Weight: 20%

4.1

Explicit Criteria to Reduce False Positives

Vague — Inconsistent

"Review this code for issues. Be conservative and only flag things you're confident about."

Explicit — Consistent

REPORT: SQL injection, hardcoded credentials, missing auth on data-modifying endpoints

DO NOT REPORT: naming conventions, style, minor refactors, performance micro-opts

Why "Only Report High-Confidence Findings" Fails

Confidence calibration is not reliable across runs. The same code may get flagged in one run and not another. Specify categories instead: ALWAYS report / NEVER report / SOMETIMES report (with conditions).

Managing Trust Through False Positive Rates

High FPs in one category undermine trust in ALL categories. If developers dismiss 70% of security alerts as false positives, they'll miss the 30% that are real. Temporarily disable the high-FP category while fixing its criteria.

4.2

Few-Shot Prompting

When to Use Few-Shot

  • Detailed prose still produces inconsistent output format
  • The task involves ambiguous cases you need to handle a specific way
  • You need the model to generalize judgment to patterns not explicitly specified
  • Extraction tasks with varied document structures

Structure of Effective Examples

Each example should show: input, correct output, and for ambiguous cases: why this was right.

EXAMPLE 1 — SQL Injection (should report):
Input: "SELECT * FROM users WHERE id = " + user_id
Output: {severity: "CRITICAL", category: "sql_injection", ...}

EXAMPLE 2 — Internal enum (should NOT report):
Input: "SELECT * FROM config WHERE key = " + config_key
Context: config_key is a validated enum from internal config, not user input
Output: null  (not a vulnerability — input is not user-controlled)

EXAMPLE 3 — Request parameter (ambiguous → explain reasoning):
Input: "SELECT * FROM products WHERE category = " + category
Context: category comes from a request parameter
Output: {severity: "HIGH", ...}
Reasoning: Request parameters are user-controlled and must be treated as untrusted.

Few-Shot for Missing Field Handling

EXAMPLE — Missing fields:
Document: "Acme Corp billed us $1,250 for consulting services."
Output: {invoice_number: null, date: null, amount: 1250.00, vendor: "Acme Corp"}
Note: Return null for absent fields — never fabricate values.

Force the Specific Tool

response = client.messages.create(
    model="claude-opus-4-5",
    tools=[EXTRACTION_TOOL],
    # Force specific tool — don't use "auto" (Claude might respond with text)
    tool_choice={"type": "tool", "name": "extract_invoice_data"},
    messages=[...]
)

Nullable Fields — Critical for Preventing Hallucination

Wrong — Forces Hallucination

"invoice_date": {"type": "string"}

Claude WILL make up a date if none exists in the document.

Right — Allows Honest Null

"invoice_date": {"type": ["string", "null"]}

Claude returns null when not found — no fabrication.

Schema Patterns

# Enum with "other" + detail
"expense_category": {"enum": ["travel", "software", "other"]},
"expense_category_detail": {"type": ["string", "null"], "description": "Required when 'other'"}

# Confidence field
"extraction_confidence": {
    "enum": ["high", "medium", "low"],
    "description": "high: clear text; medium: inferred; low: ambiguous"
}
4.4

Validation, Retry & Feedback Loops

Retry with Specific Error Feedback

When extraction fails validation, include the specific error in the retry prompt — don't retry blindly.

def extract_with_retry(document: str, max_retries: int = 3) -> dict:
    extraction = extract_invoice(document)

    for attempt in range(max_retries):
        errors = validate_extraction(extraction)
        if not errors:
            return extraction  # Valid — done
        if attempt == max_retries - 1:
            break
        # Retry with specific error feedback
        extraction = extract_invoice_with_feedback(
            document=document,
            prior_extraction=extraction,
            validation_errors=errors  # Tell Claude exactly what was wrong
        )
    return extraction  # Return best attempt even if imperfect

When Retries Will NOT Help

Retry WILL fix

Format error: "date '01/15/2024' is not in YYYY-MM-DD format"
Claude can reformat to "2024-01-15".

Retry will NOT fix

Missing info: "invoice_number is null but required"
There's no invoice number in the document. Route to human review instead.

4.6

Multi-Instance and Multi-Pass Review

Self-Review Limitation

When Claude reviews code it just wrote, it retains the reasoning that led to those decisions. It is less likely to question choices it just made — the same cognitive bias humans have when proofreading their own work.

Multi-Pass Architecture

# Pass 1: Per-file local analysis (focused, deep)
for filename, content in files.items():
    findings = claude.create(messages=[{
        "content": f"Analyze {filename} for logic bugs, error handling gaps,
security vulnerabilities. Do NOT consider cross-file dependencies."
    }])
    per_file_findings[filename] = findings

# Pass 2: Cross-file integration (data flow, interfaces, architecture)
integration = claude.create(messages=[{
    "content": f"Given per-file analyses: {per_file_findings}
Identify ONLY cross-file issues: data flow bugs, interface mismatches,
missing error propagation, architectural violations, race conditions.
Do NOT repeat issues already in per-file analyses."
}])

Key Concepts Summary

Explicit criteria
Specify WHAT to report and NOT report — don't rely on "confidence"
Few-shot examples
Most effective for format consistency and ambiguous case handling
tool_use for structured output
Eliminates JSON syntax errors; doesn't prevent semantic errors
Nullable fields
Required for any field that may be absent — prevents fabrication
tool_choice
auto / any / forced — use forced when specific tool must run first
Validation-retry
Include specific errors in retry; retries fix format, not missing info
Batch API
50% savings, 24hr window, no SLA, no multi-turn tool calling
custom_id
Correlates batch requests to responses; use for failure resubmission
Self-review bias
Independent instance reviews better than self-review
Multi-pass review
Per-file passes (depth) + integration pass (cross-file issues)

What the Exam Will Test You On

"Your extraction pipeline returns empty strings for missing fields. Why is this a problem?"
Cannot distinguish "field not in document" from "field is empty string" — use null
"Claude is told to 'be conservative' but still reports too many false positives. Fix?"
Replace vague instruction with explicit REPORT/DO NOT REPORT category lists
"You need guaranteed structured output. Which is most reliable?"
tool_use with JSON schema — eliminates syntax errors (but not semantic errors)
"Your batch job has 47 failures. What do you do?"
Use custom_id to identify failures, analyze failure reason, resubmit with fixes (chunk if context exceeded)
"Why is reviewing code with the same Claude session that generated it less effective?"
Session retains reasoning context; independent instance reviews without that bias
