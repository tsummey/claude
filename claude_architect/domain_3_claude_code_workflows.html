3.1

CLAUDE.md Configuration Hierarchy

The Three Levels

USER
~/.claude/CLAUDE.md
Personal only, NOT version controlled. Only applies to YOUR sessions. Teammates never see this.
PROJECT
project/.claude/CLAUDE.md  or  project/CLAUDE.md
Version controlled, team-wide. All developers who clone the repo get this. This is where team standards live.
DIRECTORY
project/src/CLAUDE.md
Applies when working in that directory. Overrides project-level for files in that subdirectory.
#1 Team Configuration Mistake

A developer puts team standards in ~/.claude/CLAUDE.md. New teammates clone the repo — they don't get the standards. Claude behaves differently for every developer. Fix: Move standards to project-level .claude/CLAUDE.md. Diagnose with /memory command.

The @import Syntax

# In packages/billing/CLAUDE.md
@import ../../docs/shared-standards.md
@import ../../docs/billing-specific-rules.md
@import ../../docs/pci-compliance-rules.md
# Note: NOT importing frontend-standards.md — not relevant here

3.3

Path-Specific Rules

CLAUDE.md files are directory-bound. If test files are scattered throughout the codebase, use .claude/rules/ with YAML frontmatter to apply rules by file type across many directories.

---
paths:
  - "**/*.test.tsx"
  - "**/*.test.ts"
  - "**/__tests__/**/*"
---

# Testing Conventions
- One describe block per component/function
- Use it() not test() for consistency
- Arrange-Act-Assert pattern for all test cases

3.4

Plan Mode vs. Direct Execution

Direct Execution
  • Fix a specific bug (stack trace provided)
  • Add input validation to one function
  • Update a config value
  • Well-understood, clear scope
Plan Mode
  • Restructure monolith into microservices
  • Migrate library (45+ files affected)
  • Add comprehensive test coverage to legacy codebase
  • New feature with multiple valid approaches

What Plan Mode Does

Claude explores the codebase, identifies all affected areas, considers multiple approaches, and presents a structured plan. It does NOT make any changes until you approve.

The Explore Subagent

For tasks requiring extensive codebase discovery before planning, Explore runs in an isolated context — only returning a structured summary to the main session, leaving plenty of room for implementation.

Expert Workflow: Plan Mode + Direct Execution

1. Enter plan mode → understand codebase, design approach
2. Approve the plan
3. Switch to direct execution → implement the approved plan
# Each implementation step has clear scope because the plan defined it
3.5

Iterative Refinement Techniques

When Prose Fails, Use Examples

# WEAK: Prose description (interpreted inconsistently)
"Normalize the phone number to a standard format"

# STRONG: Concrete examples (unambiguous)
"""
Input: "555-123-4567"     → Output: "+15551234567"
Input: "(555) 123-4567"   → Output: "+15551234567"
Input: "not a phone"      → Output: null
"""

The Interview Pattern

Before implementing a complex feature, ask Claude to interview you first: "Before implementing this caching layer, ask me questions about requirements, edge cases, and constraints." Answer these FIRST, then implement.

Sequential vs. Parallel Issue Resolution

# Issues that INTERACT → fix all in ONE message
"Fix these two issues together. They interact: [issue 1] [issue 2]"

# Issues that are INDEPENDENT → fix sequentially
"Fix issue 1 first."
# (After fix) "Now fix issue 2."
# Sequential lets you verify each fix before the next
3.6

CI/CD Integration

The -p Flag — Non-Interactive Mode

The #1 CI/CD Requirement

Without -p, the job hangs forever waiting for user input.

# WRONG: Hangs in CI
claude "Review this PR for security issues"

# CORRECT: Non-interactive, outputs to stdout, exits
claude -p "Review this PR for security issues"

# With structured JSON output for automated processing
claude -p "Analyze the diff for security vulnerabilities" \
  --output-format json \
  --json-schema ./schemas/security-review-schema.json
Session Isolation for Code Review

Always use a fresh Claude instance to review code. A session that wrote the code retains reasoning context and is less likely to question its own decisions. In CI, this happens naturally — every job starts fresh.

Avoiding Duplicate Comments on Re-runs

claude -p "Review the NEW changes in this PR.
Prior findings below — only report NEW or UNFIXED issues.

Prior findings:
$(cat prior_findings.json)

New diff:
$(git diff main...HEAD)" \
  --output-format json
